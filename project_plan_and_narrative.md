# 底盘虚拟驾驶 Agent 产品建设规划

## 1. 重新定义项目对象

这个项目不应被泛化为“面向某类工程软件的 Agent 平台”。它更准确的对象是：

> 面向底盘虚拟驾驶试验、虚拟调参、运动平台联调、数据记录评价和仿真闭环的业务 Agent。

当前 `main.py` 已经不是一个简单 GUI，而是把底盘虚拟驾驶台架中的多个关键环节集中到了一个控制面板里：CarSim 数据集读取与写入、车型和悬架参数切换、工况与起点下发、DSpace/Simulink 触发、IMU/CarSim/MOOG/电控数据记录、视觉补偿、触感反馈、离线数据对比、评价信息入库等。

因此，Agent 的产品规划也应该从这些真实业务链路出发，而不是抽象地说“工具调用、数据分析、报告生成”。更好的建设思路是：

> 先让 Agent 接管底盘虚拟驾驶工程师每天反复做的关键动作，再逐步把这些动作串成“试验准备 - 参数调整 - 仿真/台架运行 - 数据记录 - 评价分析 - 方案沉淀”的闭环。

## 2. 当前业务底座来自哪里

基于 `main.py`，当前系统已经具备八类底盘虚拟驾驶业务能力。

### 2.1 CarSim 车型与悬架参数管理

当前代码已经通过 CarSim COM 接口读取和操作：

- `Vehicle: Assembly` 车型数据集。
- `Suspension: Spring` 弹簧数据集。
- `Suspension: Auxiliary Roll Moment` 防倾杆/辅助滚转力矩数据集。
- `Suspension: Measured Total Roll Stiffness` 实测滚转刚度数据集。

对应函数包括 `UpdateTuningParam`、`CurrentVehicleSpringPage`、`onCarChange`、`onFrontSpringChange`、`onRearSpringChange`、`onFrontAuxMChange`、`onRearAuxMChange`、`ApplyTuningParam` 等。

这说明当前产品已经具备“底盘虚拟调参”的核心入口，而不是普通的软件操作入口。

### 2.2 弹簧和防倾杆虚拟调参

当前系统支持对前/后、左/右弹簧和前/后防倾杆进行选择、百分比增减、数值修改、方案记录、方案应用和恢复默认。

对应函数包括 `add_subtract_button`、`OnFrontSpringTextChanged`、`OnRearSpringTextChanged`、`recordApplyTuningParam`、`Scheme_SetApplyTuningParam`、`resetApplyTuningParam`、`ApplyTuningParam`。

这条链路直接对应底盘调校业务里的“刚度分配、滚转控制、舒适性/操稳性取舍”。

### 2.3 工况、地图、起点与 cueing 联动

当前系统已经有 `scenarios.json` 场景配置、地图/起点选择、RFPRO 坐标接收、起点坐标下发、工况选择和 cueing 参数切换。

对应函数包括 `load_scenario_data`、`update_start_points`、`confirm_scenario_settings`、`condition_choose`、`receive_rfpro_coordinates_data`，并依赖 `cueing_change` / `cueing_change1`。

这条链路对应的是“不同道路/工况/起点下的虚拟驾驶试验准备”。

### 2.4 DSpace、Simulink 与 CarSim 仿真执行

当前系统可以触发 CarSim 运行、打开 Simulink、编译 DSpace，并读取 `LastRun.csv` 形成离线仿真结果。

对应函数包括 `RunDspace`、`OpenSimulink`、`BuildDspace`、`clear`、`viewOfflineData`。

这条链路对应的是“虚拟样车参数修改后的仿真验证”。

### 2.5 台架与驾驶模拟器实时数据记录

当前系统持续接收并记录多源数据：

- IMU 数据。
- CarSim 输入/车辆状态数据。
- MOOG 运动平台数据。
- DisusX 电控输入/输出数据。
- 视觉补偿数据。
- RFPRO 坐标数据。

对应函数包括 `start_record`、`finish_record`、`save_data_async`、`receive_imu_data`、`receive_carsim_data`、`receive_moog_data`、`receive_electrical_control_i_data`、`receive_electrical_control_o_data`、`receive_compensator_data`。

这条链路对应的是“底盘虚拟驾驶试验数据采集与可追溯归档”。

### 2.6 评价录入、参数导出与数据库归档

当前系统在记录结束后，会弹出评价信息填写窗口，支持记录评价人、评价次数、工况版本、感受版本等字段，并复制数据目录、写入 MySQL，同时可导出 CarSim 参数文件。

对应函数包括 `create_evaluation_dialog`、`save_record_data`、`save_parameters_async`。

这条链路对应的是“主观评价 + 客观数据 + 参数版本”的试验闭环。

### 2.7 离线数据读取、曲线绘制与对比分析

当前系统支持加载历史记录文件夹，读取 `CarsimData.csv`、`MoogData.csv`、`IMUData.csv`，并绘制离线曲线；也支持 CSV 对比分析和实时曲线显示。

对应函数包括 `load_record`、`load_csv_data`、`plot_loaded_data`、`run_compare_csv`、`update_plots`、`toggle_plot_visibility`。

这条链路对应的是“试验后分析、数据复盘和方案对比”。

### 2.8 运动平台、视觉补偿和触感反馈控制

当前系统支持平台控制命令、一键启动/停机、平台偏置、驾驶员操作数据回放、视觉补偿参数、视觉延迟补偿参数、方向盘触感增益和反馈力矩计算。

对应函数包括 `one_click_start`、`one_click_stop`、`sendData2PlatformControl`、`sendDataPlatformOffset2`、`send_operation_data`、`ApplyVisualCompensation`、`ApplyVisualDelayCompensation`、`save_haptic_settings`、`get_f_data`、`sendData2CarSim`。

这条链路对应的是“驾驶模拟器沉浸感、运动平台一致性和手感调校”。

## 3. 产品建设主线应从业务闭环出发

建议把后续产品建设拆成五条底盘业务闭环，而不是泛泛地拆成“工具、数据、报告、可视化”。

### 主线一：虚拟样车与底盘参数 Agent

目标是让 Agent 成为 CarSim 车型、弹簧、防倾杆、滚转刚度参数的自然语言入口。

第一步建设：

- 查询当前车型、前后弹簧、防倾杆、滚转刚度数据集。
- 根据用户指令切换车型或悬架部件。
- 支持“前弹簧提高 5%”“后防倾杆降低 10%”“恢复默认方案”等明确指令。
- 每次修改自动记录修改前后参数、修改原因和操作者。

第二步建设：

- 支持“车身侧倾大一点”“转向响应太慢”“舒适性偏硬”等底盘语言到参数建议的映射。
- 建立弹簧刚度、防倾杆、滚转刚度与舒适性、操稳性、侧倾、转向响应之间的模板规则。
- 支持多套调校方案保存、对比、回滚。

第三步建设：

- 支持“做一版偏运动调校”“做一版偏舒适调校”“在当前车型上做单移线稳定性优化”等目标型任务。
- 调用仿真、离线对比和评价数据，自动生成推荐参数组合。

### 主线二：工况与驾驶模拟试验准备 Agent

目标是让 Agent 接管试验开始前的工况、地图、起点、cueing 和平台准备动作。

第一步建设：

- 通过自然语言选择工况，例如“切到单移线”“切到中心区”“使用一阶起伏工况”。
- 选择地图和起点，并下发起点坐标。
- 自动触发 cueing 参数切换。

第二步建设：

- 建立“试验准备清单”：CarSim 工况、起点、平台状态、相机状态、记录路径、评价字段是否准备好。
- 用户说“准备做单移线试验”，Agent 自动完成工况选择、起点下发、平台检查和记录设置。

第三步建设：

- 根据 RFPRO 实时坐标和车辆状态识别是否进入记录起点/终点区域。
- 支持自动开始记录、自动结束记录、自动提示异常状态。

### 主线三：台架运行与多源数据记录 Agent

目标是把当前分散的记录按钮和数据通道，升级为面向试验任务的记录助手。

第一步建设：

- 自然语言启动/结束记录。
- 自动同步相机录制、IMU、CarSim、MOOG、电控、视觉补偿数据。
- 支持设置保存路径、是否导出参数、是否记录电控、是否自动重命名。

第二步建设：

- 记录结束后自动检查数据完整性，例如 IMU、CarSim、MOOG 是否都有数据，时间长度是否一致，关键通道是否缺失。
- 自动生成本次试验摘要：车型、工况、起点、参数方案、记录时长、关键状态。

第三步建设：

- 将记录任务和评价任务绑定，形成“一次试验”的统一对象。
- 每次试验都能追溯到参数版本、工况版本、感受版本、数据文件和评价结果。

### 主线四：仿真验证与离线分析 Agent

目标是让 Agent 在参数修改后能自动完成仿真、读取结果、对比方案，而不是只停留在“改参数”。

第一步建设：

- 支持“运行当前方案仿真”“打开 Simulink”“编译 DSpace”“清空离线结果”等指令。
- 运行后读取 `LastRun.csv`，生成关键指标摘要。
- 打开或生成 CSV 对比分析。

第二步建设：

- 把 `RunDspace` 当前已有的 AVy 结果处理升级为标准评价指标抽取。
- 针对单移线、转弯、中心区、一阶起伏等工况，定义不同指标：横向加速度、横摆角速度、侧倾角、方向盘转角、方向盘转速、平台姿态、IMU 响应等。
- 自动给出“当前方案相对上一方案”的变化说明。

第三步建设：

- 支持多轮参数扫描：修改一组弹簧/防倾杆参数，运行仿真，读取指标，保留较优方案。
- 形成“参数 - 工况 - 指标 - 主观评价”的闭环样本。

### 主线五：驾驶感与沉浸一致性 Agent

目标是把运动平台、视觉补偿和触感反馈纳入底盘虚拟驾驶体验调校。

第一步建设：

- 支持“一键启动平台”“一键停机”“设置平台偏置”“回放驾驶员操作数据”。
- 支持设置视觉补偿和视觉延迟补偿参数。
- 支持保存方向盘触感增益。

第二步建设：

- 把平台、视觉、触感参数与具体工况和车型绑定。
- 支持“这个工况下平台俯仰太强”“视觉延迟感觉明显”“方向盘太重”等体验诉求到参数建议。

第三步建设：

- 建立虚拟驾驶体验评价模板，把平台运动、视觉延迟、方向盘力矩和车辆响应统一评价。
- 让 Agent 能根据驾驶员反馈和数据曲线建议下一轮 cueing、视觉补偿或触感参数调整。

## 4. 推荐产品阶段路线

### 阶段 0：当前状态，GUI 控制台 + 初步 Agent 入口

当前项目已经具备大量底盘虚拟驾驶业务能力，但主要以 GUI 控件和函数调用形式存在。

当前阶段的核心价值是：

- 已经接入 CarSim、运动平台、IMU、MOOG、电控、视觉补偿、数据库等关键对象。
- 已经覆盖底盘调参、工况准备、记录归档、离线分析、触感反馈等核心场景。
- 已经证明在现有 GUI 上挂载 Agent 是可行入口。

当前阶段的主要问题是：

- 功能是按界面组织的，不是按试验任务组织的。
- Agent 如果只调用按钮级 action，会缺少底盘业务语义。
- 参数、记录、评价、仿真结果之间还没有形成统一任务对象。

### 阶段 1：把 GUI action 升级为底盘业务 Skill

这一阶段不急着做“万能 Agent”，而是先把 `main.py` 中已有功能抽象为业务 Skill。

优先建设的 Skill 包括：

- `车辆与悬架参数查询 Skill`：查询当前车型、弹簧、防倾杆、滚转刚度。
- `悬架参数修改 Skill`：按数值或百分比修改弹簧/防倾杆，并支持回滚。
- `工况准备 Skill`：选择工况、地图、起点并下发 cueing。
- `试验记录 Skill`：启动/结束记录，同步相机、电控、参数导出和评价弹窗。
- `离线数据加载 Skill`：加载历史记录并绘制 IMU/CarSim/MOOG 对比。
- `仿真运行 Skill`：运行 CarSim/DSpace，读取结果并导出摘要。
- `平台与触感控制 Skill`：平台启停、偏置、视觉补偿、触感增益。

阶段目标是让 Agent 能可靠完成明确业务指令，而不是只点击界面。

### 阶段 2：形成“单次底盘虚拟驾驶试验”对象

这一阶段的关键是建立统一的数据模型，把一次试验从头到尾串起来。

建议定义 `VirtualDriveTest` 对象，至少包含：

- 车型与 CarSim 数据集。
- 工况、地图、起点和 cueing 方案。
- 弹簧、防倾杆、滚转刚度参数版本。
- 平台、视觉补偿、触感反馈配置。
- 记录数据路径和文件清单。
- 评价人、评价次数、工况版本、感受版本。
- 仿真结果、离线分析结果和报告路径。

有了这个对象，Agent 才能回答底盘业务问题，例如：

- “上一轮单移线试验用了哪套后防倾杆？”
- “这次改了前弹簧以后，横向加速度和侧倾角有什么变化？”
- “把这次记录和上次运动版方案对比一下。”

### 阶段 3：建设底盘调校工作流

这一阶段开始从“单点操作”进入“多步任务”。

优先建设三条工作流：

#### 工作流 A：明确参数修改验证

用户说：“把前弹簧提高 5%，跑一轮单移线。”

Agent 执行：

1. 读取当前车型和前弹簧数据。
2. 修改前弹簧参数并记录版本。
3. 切换单移线工况和起点。
4. 运行仿真或提示开始台架试验。
5. 记录数据并检查完整性。
6. 输出前后指标对比。

#### 工作流 B：工程诉求调参

用户说：“现在侧倾有点大，帮我收一下。”

Agent 执行：

1. 判断诉求属于滚转控制问题。
2. 查询当前弹簧、防倾杆和滚转刚度配置。
3. 按模板建议提高前/后防倾杆或调整弹簧刚度。
4. 给出调整幅度和风险提示。
5. 用户确认后写入 CarSim。
6. 运行指定工况验证。

#### 工作流 C：试验记录与报告

用户说：“准备记录一轮中心区试验，结束后生成评价摘要。”

Agent 执行：

1. 切换中心区工况和起点。
2. 检查 IMU、CarSim、MOOG、电控、相机记录状态。
3. 开始记录。
4. 结束后弹出或自动补全评价信息。
5. 保存数据、导出参数、写入数据库。
6. 生成试验摘要和曲线对比。

### 阶段 4：建设指标体系和评价模板

这一阶段要把“底盘虚拟驾驶”做实，关键是指标不要泛化。

建议先围绕当前代码已经记录的信号建设指标：

- 车身姿态：roll、pitch、yaw。
- 车辆运动：vel_x、acc_x、acc_y、acc_z。
- 角速度与角加速度：ang_x、ang_y、ang_z、ang_acc_x、ang_acc_y、ang_acc_z。
- 驾驶输入：steering_angle、steering_speed、throttle、pbk_con。
- 平台响应：MOOG 位置、姿态、速度、加速度。
- 电控信号：DisusX 输入/输出。
- 视觉补偿：视觉补偿回传数据。
- 触感反馈：方向盘力矩、摩擦、阻尼、回正、限位贡献。

评价模板建议按工况拆：

- 单移线：横向加速度峰值、侧倾角峰值、方向盘峰值、横摆响应、平台跟随。
- 转弯：稳态侧倾、横向加速度、方向盘力矩、舒适性反馈。
- 中心区：方向盘中心区力矩、回正特性、转角/力矩线性。
- 一阶起伏：俯仰、垂向加速度、平台响应、视觉延迟感。

Agent 后续的分析能力应围绕这些指标展开，而不是只做通用 CSV 读取。

### 阶段 5：目标驱动调校与优化

当阶段 1-4 稳定后，才能进入目标驱动优化。

建议先从有限目标开始：

- 降低单移线侧倾峰值。
- 降低一阶起伏垂向加速度峰值。
- 优化中心区方向盘力矩线性。
- 在不显著降低舒适性的前提下提升转弯支撑。

Agent 的优化流程应是：

1. 读取当前参数方案。
2. 根据目标选择可调参数，例如前/后弹簧、前/后防倾杆、触感增益、cueing 参数。
3. 在安全范围内生成候选方案。
4. 运行仿真或提示台架试验。
5. 抽取指标并与基线对比。
6. 保留较优方案，解释为什么更优。
7. 归档参数、指标和用户评价。

这一步才是从“操作型 Agent”升级为“底盘调校 Agent”的关键。

### 阶段 6：自进化与经验沉淀

自进化不能直接从“让 Agent 自己改工具”开始，而要从底盘业务经验沉淀开始。

建议沉淀四类记忆：

- 参数记忆：不同车型、工况、调校目标下的参数组合。
- 结果记忆：每次参数变化带来的指标变化。
- 评价记忆：驾驶员主观评价和客观指标之间的关系。
- 流程记忆：哪些试验准备步骤、数据检查步骤最容易出错。

当这些记忆稳定后，再引入 Hermes 类能力：

- 自动总结高频任务并生成新 Skill。
- 根据失败记录修正工具调用顺序。
- 根据用户评价优化调参模板。
- 通过评测集验证新策略是否可靠。

## 5. 下一阶段最应该做什么

如果要让项目更快从 Demo 变成产品，下一阶段建议优先做四件事。

### 5.1 建立底盘业务 Skill 清单

不要把所有函数都注册成 Agent tool。应该从 `main.py` 中挑出高阶业务入口，形成第一批 Skill。

优先级最高的是：

- 查询当前车型与悬架配置。
- 修改弹簧/防倾杆参数。
- 应用/保存/回滚调参方案。
- 选择工况、地图和起点。
- 启动/结束记录。
- 运行仿真并读取结果。
- 加载历史记录并对比曲线。

### 5.2 建立一次试验的数据模型

把记录文件夹、参数导出、评价字段、工况、车型、起点、数据文件、仿真结果绑定到同一个 `VirtualDriveTest`。

这个模型是后续报告自动化、历史检索、调参优化和自进化的地基。

### 5.3 建立底盘指标与评价模板

围绕单移线、转弯、中心区、一阶起伏四类当前已有工况，先定义最小指标集。

有了指标，Agent 才能从“我帮你改了参数”升级为“我能判断这次改得怎么样”。

### 5.4 做三条可演示端到端流程

建议先做：

- “修改悬架参数 -> 运行仿真 -> 输出指标对比”。
- “准备工况 -> 开始记录 -> 结束归档 -> 生成试验摘要”。
- “加载两次记录 -> 对比 IMU/CarSim/MOOG 曲线 -> 给出差异说明”。

这三条流程最能体现当前项目的底盘虚拟驾驶针对性。

## 6. 推荐汇报话术

### 30 秒版本

当前项目不是泛泛地做一个 Agent 平台，而是围绕底盘虚拟驾驶台架的真实业务链路来建设 Agent。现有系统已经接入 CarSim 车型和悬架参数、工况起点、DSpace/Simulink、IMU/CarSim/MOOG/电控数据记录、视觉补偿、触感反馈和评价入库。下一步不是继续堆按钮，而是把这些能力抽象成底盘业务 Skill，再串成“参数修改 - 工况准备 - 仿真/台架运行 - 数据记录 - 指标评价 - 方案沉淀”的闭环。

### 2 分钟版本

我们现在的基础不是一个空白 GUI，而是已经有一套底盘虚拟驾驶控制台。它能操作 CarSim 车型、弹簧、防倾杆和滚转刚度，能选择工况、地图和起点，能联动 cueing、平台、相机、视觉补偿和触感反馈，也能记录 IMU、CarSim、MOOG、电控等多源数据，并在记录后进行评价入库和离线对比。

所以这个 Agent 的建设路线应该非常贴近底盘业务。第一步，把当前 GUI 里的关键业务动作抽象成 Skill，例如悬架参数修改、工况准备、试验记录、仿真运行、离线分析。第二步，建立“单次底盘虚拟驾驶试验”对象，把车型、参数、工况、数据、评价和结果统一起来。第三步，围绕单移线、转弯、中心区、一阶起伏等工况建立指标模板，让 Agent 不只是会操作，还能判断方案变化。第四步，再做目标驱动调校，例如降低侧倾、优化中心区手感、改善一阶起伏舒适性。这样项目才能从 GUI 操作助手，逐步升级为真正的底盘虚拟驾驶调校 Agent。

## 7. 一页 PPT 表达建议

一页 PPT 可以用“当前底座 -> 产品化路径 -> 下一步交付”三段式表达。

左侧：当前已有底盘业务底座

- CarSim 车型/弹簧/防倾杆/滚转刚度。
- 工况、地图、起点、cueing。
- DSpace、Simulink、CarSim 仿真。
- IMU、CarSim、MOOG、电控、视觉补偿记录。
- 评价入库、离线曲线对比、触感反馈。

中间：Agent 产品化路径

- 业务 Skill 化。
- 单次试验对象化。
- 工况指标模板化。
- 调参工作流自动化。
- 目标驱动优化。
- 经验沉淀与自进化。

右侧：下一阶段重点交付

- 悬架参数修改 Skill。
- 工况准备 Skill。
- 试验记录 Skill。
- 仿真与离线分析 Skill。
- `VirtualDriveTest` 数据模型。
- 三条端到端演示流程。

底部结论：

> 当前项目已经具备底盘虚拟驾驶业务底座，下一阶段的核心不是增加通用 Agent 能力，而是把现有底盘业务函数产品化为可编排、可追溯、可评价、可优化的 Agent 工作流。
