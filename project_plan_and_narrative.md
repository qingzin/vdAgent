# 底盘虚拟驾驶全域智能体项目计划与陈述逻辑

## 1. 项目总目标

本项目的整体目标不是简单做一个 GUI，也不是只在某个软件里增加一个聊天框，而是构建一个面向底盘虚拟驾驶业务的全域智能体。

这个智能体需要能够联动底盘虚拟开发链路中的关键软件和数据系统，理解用户的自然语言需求，自动完成参数读取、参数修改、仿真调用、结果分析、评价记录、报告生成等任务，并逐步具备多步自主执行和持续进化能力。

更准确的项目定位是：

> 构建面向底盘虚拟驾驶场景的全域智能体，实现从“用户意图”到“软件联动、参数调整、仿真执行、数据分析、报告产出”的智能闭环。

## 2. 是否只是做一个 GUI

不是。

GUI 只是当前阶段的入口和验证载体，不是项目的最终形态。

当前采用 GUI 挂载方式，是因为现有业务能力已经集中在自研软件界面中，通过对话框挂载 Agent 可以快速验证自然语言操作的价值，低成本打通第一批控制链路。

但长期来看，如果 Agent 只依赖 GUI 页面上的按钮和控件，它会受到明显限制：

- 只能执行页面上已经暴露的功能。
- 很难跨软件联动。
- 很难完成多步复杂任务。
- 很难沉淀用户经验和业务流程。
- 很难支撑自适应调参、自动分析、报告生成和持续进化。

因此，项目后续应从“GUI 挂载式助手”升级为“跨软件、跨流程、可进化的业务智能体”。

## 3. 需要联动的软件与系统

当前和后续重点联动对象包括：

- 自研控制软件
  当前 Agent 已经以对话框形式挂载到该软件中，调用其 GUI 页面和底层业务逻辑。

- CarSim
  用于车辆模型、底盘参数、工况、仿真运行和结果导出。

- 自研控制/算法组件
  用于底盘控制、参数优化、运动风格调教、自适应调整等能力。

- 数据文件与记录系统
  包括 IMU、CarSim、MOOG、电控、视觉补偿等记录数据。

- 数据库与报告系统
  用于评价信息录入、记录归档、报告生成和历史追溯。

- 后续可扩展系统
  包括语音入口、可视化看板、低代码配置平台、自动评测系统和自进化模块。

## 4. 我们参与进去后的负责目标

在整个“底盘虚拟驾驶全域智能体”目标中，我们当前负责的核心目标可以定义为：

> 建设底盘虚拟驾驶智能体的第一阶段原型和后续可演进架构，先通过 GUI 挂载方式打通自研软件和 CarSim 的自然语言操作链路，再逐步将参数调整、仿真执行、数据分析、评价录入、报告生成等能力封装为可复用 skills 和 services，最终支撑 Agent 自主多步执行和持续进化。

换句话说，我们不是只交付一个聊天框，而是在建设智能体底座：

- 第一阶段验证 Agent 能否理解用户指令并操作现有软件。
- 第二阶段将操作能力沉淀为标准工具和服务。
- 第三阶段将复杂业务流程封装为 skills。
- 第四阶段引入优化组件、记忆系统和自进化闭环。
- 最终形成可跨软件、可多步执行、可持续变好的底盘虚拟驾驶智能体。

## 5. 功能端建设阶段

功能端可以分为四个逐级增强阶段。

### 阶段 1：指令型操作

用户输入明确指令，例如：

- “把前弹簧刚度改到 30000。”
- “切换到某个车型。”
- “开始记录。”
- “运行 CarSim 仿真。”

Agent 执行逻辑：

```mermaid
flowchart LR
    User["用户明确指令"]
    Understand["Agent 理解参数和动作"]
    Tool["调用对应工具"]
    Action["修改软件参数或触发操作"]
    Result["返回执行结果"]

    User --> Understand --> Tool --> Action --> Result
```

阶段目标：

- 验证自然语言操作可行。
- 打通 GUI action 与 Agent tool。
- 保证基础操作安全可控。

当前项目状态：

- 当前已基本处于这一阶段。
- 已经可以通过 Agent 调用部分 GUI 页面操作。
- 已经具备基本 action registry 和执行确认机制。
- 客户/对方团队对第一阶段效果评价较好，说明方向可行。

### 阶段 2：诉求型操作

用户不再给出明确数值，而是提出工程诉求，例如：

- “把悬架调软一些。”
- “让这个车更舒适一点。”
- “这个工况下侧倾有点大，帮我收一下。”

Agent 执行逻辑：

```mermaid
flowchart LR
    User["用户工程诉求"]
    Intent["Agent 理解意图"]
    Template["调用调参模板"]
    Params["生成建议参数"]
    Execute["修改参数并记录"]
    Feedback["反馈调整依据"]

    User --> Intent --> Template --> Params --> Execute --> Feedback
```

阶段目标：

- 将“工程语言”映射为“参数调整策略”。
- 建立常用调参模板。
- 让 Agent 不只是执行命令，而是理解用户诉求。

需要建设：

- 调参模板库。
- 参数影响规则。
- 常用工程表达映射。
- 参数修改记录。
- 调整前后对比。

### 阶段 3.1：复杂想法型任务

用户提出相对复杂但仍然有明确方向的目标，例如：

- “完成一个运动风格的底盘调教。”
- “基于当前车型做一版偏舒适的方案。”
- “帮我在这个工况下做一轮稳定性优化。”

Agent 执行逻辑：

```mermaid
flowchart LR
    User["复杂调教想法"]
    Plan["Agent 拆解任务"]
    Skill["调用业务 Skill"]
    Optimize["调用优化组件"]
    Sim["运行仿真/分析结果"]
    Update["迭代修改参数"]
    Report["输出方案和结论"]

    User --> Plan --> Skill --> Optimize --> Sim --> Update --> Report
```

阶段目标：

- Agent 能够完成多步任务。
- 能够调用优化组件。
- 能够根据仿真结果继续调整参数。
- 能够输出方案说明。

需要建设：

- 运动风格、舒适风格、稳定性等调教 skills。
- 参数优化组件接口。
- 仿真结果分析组件。
- 多轮迭代任务管理。
- 方案记录和报告输出。

### 阶段 3.2：目标型自适应任务

用户只给出目标，不指定具体方法，例如：

- “适应不同路况自适应完成调整。”
- “让车辆在多种工况下都保持较好的舒适性和稳定性。”
- “自动找到一组兼顾运动性和舒适性的底盘参数。”

Agent 执行逻辑：

```mermaid
flowchart LR
    Goal["用户目标"]
    Agent["Agent 自主规划"]
    Env["识别路况/工况"]
    Search["搜索/优化参数"]
    Eval["多工况评估"]
    Iterate["持续迭代"]
    Archive["归档最优策略"]

    Goal --> Agent --> Env --> Search --> Eval --> Iterate --> Archive
```

阶段目标：

- Agent 从“执行工具”升级为“目标驱动的自主系统”。
- 能够跨工况评估。
- 能够自适应调整。
- 能够沉淀最优策略和经验。

需要建设：

- 目标函数定义。
- 多工况评测集。
- 优化算法组件。
- 参数搜索和约束系统。
- 历史经验记忆。
- 安全边界和人工审核机制。

## 6. 界面端建设阶段

界面端也可以分为三个阶段。

### 阶段 1：内嵌到某一个自研软件

形态：

- Agent 以对话框形式嵌入现有自研软件。
- 主要调用单一软件内部功能。
- 适合验证自然语言操作和基础工作流。

当前项目状态：

- 当前 vdAgent 正处于这一阶段。
- 已经实现 Agent 挂载到 GUI。
- 已经能调用部分页面操作。

价值：

- 成本低。
- 见效快。
- 不破坏现有软件结构。
- 适合作为第一阶段成果展示。

局限：

- 依赖 GUI。
- 受限于单一软件。
- 很难完成跨软件任务。

### 阶段 2：跨软件调用

形态：

- Agent 仍可以以对话框出现。
- 但不只调用当前 GUI，而是能联动自研软件、CarSim、控制组件、数据系统和报告系统。

能力：

- 一句话完成跨软件操作。
- 自动读取 CarSim 参数。
- 调用控制软件修改参数。
- 运行仿真。
- 读取结果。
- 生成评价和报告。

关键变化：

- Agent 不再只是 GUI 附属。
- Agent 后面需要有统一工具网关和业务服务层。
- 每个软件只是 Agent 可调用的能力之一。

### 阶段 3：低代码平台

形态：

- 面向工程人员提供低代码/拖拉拽式配置界面。
- 用户可以配置工具、流程、模板、报告字段和分析指标。
- 降低 Agent 使用和扩展门槛。

能力：

- 拖拽配置试验流程。
- 拖拽配置报告模板。
- 配置自然语言指令与业务流程的映射。
- 配置不同车型、工况、路况的调参策略。

价值：

- 让非研发人员也能配置 Agent。
- 让 skills 和 workflows 可视化。
- 提高推广和复用能力。

## 7. 总体阶段拆分

可以将项目整体拆成六个阶段。

| 阶段 | 名称 | 核心目标 | 当前状态 |
| --- | --- | --- | --- |
| 阶段 1 | GUI Agent 原型 | 嵌入单一自研软件，完成明确指令操作 | 当前阶段，已实现初步成果 |
| 阶段 2 | 工具治理与跨软件接口 | 将 GUI action 收敛为标准工具，接入 CarSim 和控制软件 | 下一阶段 |
| 阶段 3 | 诉求理解与模板调参 | 将工程诉求映射到调参模板 | 规划阶段 |
| 阶段 4 | Skills 与优化组件 | 支持复杂调教任务和多步自主执行 | 规划阶段 |
| 阶段 5 | 数据分析与报告自动化 | 支持离线分析、评价录入、报告生成 | 规划阶段 |
| 阶段 6 | 自适应与自进化 | 支持目标驱动、自适应调参和持续进化 | 长期目标 |

汇报时建议按以下顺序展开。

### 9.1 先讲大目标

推荐话术：

> 我们不是单纯做一个 GUI，也不是做一个聊天机器人，而是希望形成一个底盘虚拟驾驶全域智能体。这个智能体未来要能联动自研控制软件、CarSim、数据记录系统和优化组件，把用户的工程语言转化为可执行的软件操作和调参流程。

### 9.2 再讲为什么先做 GUI 挂载

推荐话术：

> 第一阶段选择挂载在现有 GUI 中，是因为现有业务能力已经集中在自研软件里。通过这种方式，我们可以最低成本验证自然语言控制是否可行，也能快速让用户看到效果。

### 9.3 接着讲当前成果

推荐话术：

> 当前我们已经完成第一阶段原型：Agent 能够嵌入 GUI，以对话框形式接收用户输入，理解明确指令，并调用部分页面功能完成参数修改、记录、仿真等操作。这个阶段已经证明方向可行，对方评价也比较满意。

### 9.4 然后讲当前局限

推荐话术：

> 但我们也认识到，如果 Agent 只是依赖 GUI 页面，它会很受限。它只能调用页面上已有的功能，很难跨软件，很难做复杂任务，也很难从用户使用中持续进化。

### 9.5 最后讲未来规划

推荐话术：

> 所以下一阶段要把能力从 GUI action 升级为 tools、services、skills 和 workflows。明确指令类任务由工具直接执行；工程诉求类任务调用调参模板；复杂目标类任务调用优化组件和仿真分析；长期则通过记忆、评测和人工审核机制，让 Agent 根据用户使用不断进化。
