# 底盘虚拟驾驶全域智能体项目计划与陈述逻辑

## 1. 项目定位

本项目不应被定义为“做一个简单 GUI”或“给某个软件加一个聊天框”，而应定位为：

> 面向底盘虚拟开发、仿真调校、数据分析、评价报告和跨软件协同的全域智能体底座。

当前阶段采用“挂载在既有 GUI 窗口上的 Agent”作为切入点，是为了用最小集成成本快速验证用户价值：让用户通过自然语言完成软件界面上的常用操作、参数修改和流程触发。

但长期目标不应停留在 GUI 自动化。GUI 操作只是 Agent 的一个执行通道。更理想的形态是将 CarSim、自研控制软件、仿真数据、参数记录、评价表格、报告生成、优化算法和专家经验封装为可复用的能力单元，使 Agent 能够理解目标、拆解任务、调用工具、执行多步流程，并基于结果继续迭代。

## 2. 行业与业务背景

底盘虚拟开发场景天然具有“多软件、多参数、多工况、多评价指标、多轮迭代”的特点。实际工作中，工程师往往需要在仿真软件、控制软件、数据文件、评价表格和报告模板之间频繁切换。

当前典型痛点包括：

- 软件与流程割裂：CarSim、自研控制软件、数据分析工具、报告工具之间缺少统一任务入口。
- 操作链条长：参数修改、仿真运行、结果读取、指标评价、报告生成需要多步人工操作。
- 专家经验难沉淀：例如“把悬架调软一些”“做运动风格调校”这类工程诉求背后包含经验模板，但目前难以标准化复用。
- 数据闭环不足：仿真参数、执行记录、评价结果和优化过程没有形成统一归档。
- 自动化深度有限：传统脚本通常解决单点操作，难以根据执行结果自主推进下一步。

因此，本项目的核心价值不是替代某个按钮或脚本，而是把工程师的“意图、流程、工具、数据、经验”连接起来，形成底盘虚拟开发的智能协同入口。

## 3. 总体目标

项目总体目标是构建“底盘虚拟驾驶全域智能体”，使其能够联动 CarSim、自研控制软件及相关数据分析、评价和报告系统，支撑从简单操作到复杂调校任务的智能化执行。

长期目标可以概括为四句话：

- 让用户用自然语言表达操作、诉求或目标，而不是手动记忆软件路径和流程。
- 让 Agent 能够调用跨软件、跨数据、跨报告的能力，而不是只操作一个 GUI。
- 让工程经验沉淀为可复用的 Skills、模板、工作流和评价规则。
- 让系统具备持续学习和自我改进能力，逐步向 Hermes 类自进化 Agent 框架演进。

## 4. 我们承担的建设目标

在整体项目中，我们参与的目标不是单纯开发界面，而是建设智能体能力底座的早期版本，并逐步把现有 GUI 操作能力升级为面向业务任务的智能执行体系。

当前承担范围可以分为三层：

- 第一层是交互入口：在自研软件中以对话框或侧边栏形式接入 Agent，降低用户使用门槛。
- 第二层是执行能力：把软件操作、参数修改、数据读取、记录存档、绘图分析、报告生成等能力封装为工具或 Skills。
- 第三层是智能编排：让 Agent 能够基于用户目标选择能力、拆解步骤、执行任务、检查结果并继续推进。

这三层关系决定了项目路线：先用 GUI Agent 验证价值，再把能力从 GUI 操作中抽象出来，最后形成跨软件、跨流程、可进化的 Agent 平台。

## 5. 能力成熟度模型

建议用“能力成熟度”来陈述项目阶段，而不是简单按功能清单罗列。

### L1：指令执行型 Agent

用户输入明确指令，例如“把 XX 参数改为 YY”。Agent 理解指令后，调用对应工具完成参数定位、修改和确认。

这一阶段解决的是“能不能替用户完成明确操作”的问题。

### L2：意图理解型 Agent

用户输入工程诉求，例如“把悬架调软一些”“让转向更稳一点”。Agent 不只执行字面命令，而是调用预设规则、经验模板或参数映射关系，把模糊诉求转化为具体参数调整建议。

这一阶段解决的是“能不能理解工程语言”的问题。

### L3：任务编排型 Agent

用户输入复杂任务，例如“完成一版运动风格底盘调校”。Agent 能够拆解任务，调用参数模板、仿真运行、指标评价、数据分析和报告生成等多个 Skills，形成多步工作流。

这一阶段解决的是“能不能完成一条工程流程”的问题。

### L4：目标优化型 Agent

用户输入目标，例如“在舒适性不明显下降的前提下提升操稳表现”。Agent 能够调用优化组件，在多轮仿真和评价结果之间迭代，自动寻找更优参数组合。

这一阶段解决的是“能不能基于目标自主迭代”的问题。

### L5：自适应进化型 Agent

Agent 具备长期记忆、评价反馈、经验沉淀和能力自更新机制，能够根据用户使用、任务结果和历史案例持续优化策略，逐步接近 Hermes 类自进化 Agent。

这一阶段解决的是“能不能越用越懂业务、越用越有效”的问题。

## 6. 软件与界面演进路径

界面形态不应被视为项目本体，而应视为 Agent 能力的入口。

### 阶段 A：单软件内嵌

Agent 以内嵌对话框形式出现在某一个自研软件中，主要服务单软件场景。当前项目正处于这一阶段。

价值是验证用户接受度、验证自然语言操作是否可行、验证 GUI 自动化与软件功能能否打通。

### 阶段 B：跨软件调用

Agent 仍可以保持对话入口，但背后不再只控制一个 GUI，而是能联动 CarSim、自研控制软件、文件系统、数据分析脚本和参数管理模块。

价值是把“操作助手”升级为“流程助手”。

### 阶段 C：工程闭环

Agent 能够把参数、仿真、结果、评价、报告和归档串成闭环。用户关注目标和结果，系统负责执行过程和记录沉淀。

价值是把“流程助手”升级为“工程协同 Agent”。

### 阶段 D：低代码与组织级复用

通过低代码平台、可视化流程编排、模板管理和 Skill 市场，降低新流程接入门槛，让更多业务人员可以配置和复用 Agent 能力。

价值是把“项目工具”升级为“组织级智能体平台”。

## 7. 当前项目阶段判断

当前项目处于“阶段 A + 能力 L1”的验证期，已经完成了从概念到可演示原型的重要一步。

当前已验证的内容包括：

- Agent 可以以内嵌方式挂载到既有 GUI 中。
- 用户可以通过对话方式触发部分 GUI 操作。
- 系统已具备基础工具注册、意图识别和动作执行链路。
- 初步覆盖了参数修改、平台操作、记录存档、绘图分析等方向的能力雏形。
- 第一阶段演示反馈较好，说明用户对这种交互方式和效率提升有明确认可。

当前阶段的意义在于证明方向成立，而不是证明最终形态已经完成。

当前尚未充分验证的内容包括：

- 跨软件任务编排能力。
- 面向工程诉求的模板化调参能力。
- 仿真结果驱动的自动优化能力。
- 参数、结果、评价和报告的一体化闭环。
- 长期记忆、经验沉淀和自进化机制。

因此，对外陈述时建议明确表达：我们已经完成第一阶段原型验证，当前正从“GUI 操作型 Agent”向“工程任务型 Agent”升级。

## 8. 核心建设主线

后续建设不建议按单个功能孤立推进，而应按几条主线并行演进。

### 主线一：工具治理与安全边界

把当前 action/tool 进行重新梳理，区分底层 GUI 操作、业务功能工具和 Agent 可直接调用的高阶 Skill。

重点包括：

- 只把稳定、可解释、可回滚、有业务语义的能力暴露给 Agent。
- 对参数写入、仿真运行、文件覆盖、平台操作等高风险动作设置确认机制。
- 建立工具说明、输入输出格式、错误处理和日志记录规范。

### 主线二：业务能力 Skill 化

把常见工程操作从“点击某个按钮”升级为“完成某个业务动作”。

典型 Skill 包括：

- 参数读取与修改。
- 参数组保存与版本对比。
- CarSim 工况配置与仿真触发。
- 自研控制软件参数同步。
- 仿真结果读取与指标提取。
- 评价表格录入。
- 报告自动生成。
- 常用调校模板调用。

### 主线三：工作流编排

把多个 Skills 组合为可复用的工程工作流。

典型工作流包括：

- 单参数修改后自动运行仿真并生成结果摘要。
- 根据舒适性或操稳诉求调用模板完成一轮调参。
- 根据目标指标进行多轮参数搜索和结果对比。
- 自动读取离线数据、生成评价表格并输出报告。

### 主线四：数据与知识闭环

建立参数、仿真、评价、报告和用户反馈之间的统一记录。

重点包括：

- 每次参数修改有来源、原因、版本和执行记录。
- 每次仿真有工况、参数、结果和评价指标。
- 每次报告可追溯到对应数据和参数版本。
- 用户反馈可以反哺模板、规则和优化策略。

### 主线五：自进化 Agent 底座

参考 Hermes 类自进化 Agent 的思想，逐步引入任务记忆、执行反思、能力生成、策略优化和评测机制。

这里需要注意，自进化不应一开始就追求“自动改自己”，而应先建立可评测、可追踪、可回滚的能力演进机制。

## 9. 阶段路线图

### 第一阶段：GUI Agent 原型验证

目标是验证自然语言交互和 GUI 自动化的可行性。

交付物包括：

- 内嵌式对话入口。
- 基础工具注册机制。
- 明确指令到 GUI 操作的执行链路。
- 核心演示场景。
- 用户反馈闭环。

当前项目已基本处于该阶段末期。

### 第二阶段：工具标准化与能力抽象

目标是把“能点按钮”升级为“能调用业务能力”。

交付物包括：

- 工具分层和注册规范。
- 高阶业务 Skill 列表。
- 参数读写、数据读取、记录存档等核心能力模块。
- 执行日志、权限确认和错误恢复机制。

### 第三阶段：跨软件联动

目标是实现 CarSim、自研控制软件和数据分析模块之间的协同。

交付物包括：

- CarSim 连接与任务触发能力。
- 自研控制软件参数同步能力。
- 跨软件任务状态管理。
- 统一任务上下文和执行记录。

### 第四阶段：工程工作流自动化

目标是让 Agent 能完成完整工程任务，而不是只完成单步操作。

交付物包括：

- 常用调校模板。
- 离线数据读取与分析流程。
- 评价表格便捷录入流程。
- 报告自动生成流程。
- 参数调整、仿真、评价、报告的一体化闭环。

### 第五阶段：目标驱动优化

目标是让 Agent 能围绕工程目标进行多轮迭代。

交付物包括：

- 优化组件接入。
- 多目标评价规则。
- 参数搜索与结果对比机制。
- 自动生成调校建议。

### 第六阶段：低代码与自进化平台

目标是让业务人员可以配置流程，让 Agent 可以持续沉淀经验。

交付物包括：

- 低代码流程编排界面。
- Skill 模板管理。
- 语音交互和可视化看板。
- 任务记忆与案例库。
- 评测驱动的能力改进机制。

## 10. 推荐陈述逻辑

对外汇报时，建议不要从“我们做了一个 GUI”开始，而是从业务愿景和阶段验证开始。

### 30 秒版本

本项目的目标不是做一个简单 GUI，而是建设面向底盘虚拟开发的全域智能体。它未来要联动 CarSim、自研控制软件、数据分析、参数存档、评价表格和报告系统，把工程师的自然语言目标转化为可执行的多步任务。当前我们处于第一阶段，已经通过内嵌 GUI Agent 验证了自然语言操作和软件联动的可行性，用户反馈较好。下一步会从 GUI 操作升级为 Skill 化能力和跨软件工作流，逐步形成底盘虚拟驾驶智能体底座。

### 2 分钟版本

底盘虚拟开发的核心问题不是缺少单点工具，而是软件、参数、仿真、数据、评价和报告之间缺少统一的智能协同入口。工程师现在需要在 CarSim、自研控制软件、数据表格和报告模板之间反复切换，大量经验依赖人工记忆和手工操作。

因此，我们把项目定位为“底盘虚拟驾驶全域智能体”。短期先通过内嵌式 GUI Agent 验证用户能否用自然语言完成软件操作和参数修改；中期把参数读写、仿真触发、数据分析、评价录入、报告生成等能力封装为 Skills；长期再引入目标优化、任务记忆、低代码编排和自进化机制，使 Agent 能够根据用户目标自主完成复杂调校任务。

当前项目已经完成第一阶段原型验证，证明这种交互方式可被用户接受，也证明现有软件具备接入 Agent 的基础。后续重点不是继续堆更多 GUI action，而是进行工具治理、能力抽象和跨软件流程编排，把当前原型升级为可扩展、可复用、可持续演进的智能体平台。

## 11. 一页 PPT 建议结构

如果整理成一页项目管理式 PPT，建议按以下逻辑排版：

- 左上：项目愿景  
  “构建底盘虚拟驾驶全域智能体，联动软件、数据、流程和专家经验。”

- 左下：当前阶段  
  “阶段一：单软件内嵌 GUI Agent 原型验证已完成，用户反馈满意。”

- 中间：能力演进路径  
  “指令执行 -> 意图理解 -> 任务编排 -> 目标优化 -> 自适应进化。”

- 右上：未来能力地图  
  “CarSim、自研控制软件、离线数据、参数存档、评价表格、报告生成、语音交互、可视化、低代码平台。”

- 右下：下一阶段重点  
  “工具分层治理、业务 Skill 化、跨软件联动、工程工作流闭环。”

这页 PPT 的核心表达应是：当前不是终点，而是一个正确切口；我们已经验证了入口价值，下一步要建设真正支撑底盘虚拟驾驶业务的智能体能力底座。

## 12. 管理层关心的关键结论

- 项目目标清晰：不是 GUI 工具，而是底盘虚拟开发智能体平台。
- 当前阶段合理：先做单软件内嵌原型，是最低风险、最快验证价值的路径。
- 用户反馈积极：第一阶段说明交互方式和业务切口成立。
- 后续路线明确：从工具调用走向 Skill 化、流程化、跨软件、目标优化和自进化。
- 建设重点可控：近期应优先做工具治理、能力抽象、数据闭环和核心工作流，而不是盲目增加零散 action。
